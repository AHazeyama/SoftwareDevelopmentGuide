# Overview
要件定義で決定した内容を基に、システム全体の構成と各機能の実現方法を定義します。  
要件を満たすための機能を整理し、体系的に表現する事が目的です。
# Oparation
基本設計書は、発注者と開発者が共通認識を持つための文書です。  
内容は専門用語を避け、ITの知識がなくても理解できる表現を心掛けます。  
また、要件定義の内容が漏れていたり、誤解していないかを確認する役割も担っています。  
システムの全体像が分かる構成にする事も重要です。  
設計内容を変更した場合は文書を更新し、関係者によるレビューと合意を経て反映します。  
要件定義書との整合性を継続的に確認し、要件漏れや仕様変更がないことを保証します。  
# Content
基本設計書に含まれる主な構成要素、システム規模によって、以下のドキュメント群で構成されます。  
```
　凡例
　　🟥 必須：要件定義で必ず記載すべき項目  
　　🟨 推奨：プロジェクト規模に応じて記載  
　　🟩 必要に応じて：あるとなお良い
```
🟥 システム構成  
　システムを構成する技術に対して適用する製品を記載し、選定理由も記載します。  
　・OS  
　・クラウド  
　・DB  
　・ネットワーク  
　・ミドルウェア  
🟥 機能概要  
　必要機能を列挙し、主な用途と注意点も記載します。  
　フローは図を使用し、視覚的に分かりやすい構成を意識して記載します。  
　・機能一覧  
　・業務フロー  
　・処理フロー  
🟥 UI設計  
　各部品は彩色し、視覚的に用途が判別できる事が望ましいです。  
　図は簡潔にして、後述する部品説明との関連が明確に分かることを意識して設計します。  
　・画面一覧  
　・画面遷移  
　・画面レイアウト  
　・帳票  
🟥 データ設計  
　概念的な構図で関連性が分かる図で表します。
　やり取りされるデータの内容が分かる図であることも重要です。
　・ER図  
　・データ項目  
　・コード体系  
🟥 外部連携  
　連携手段を明確にし、利点のみではなく、問題点も記載してあると親切です。  
　・API  
　・CSV  
　・FTP  
　・ETL  
　・メール
