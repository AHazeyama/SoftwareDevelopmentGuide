# Overview  
要件定義は、システム開発における最も重要な工程です。  
「何を実現するシステムなのか」を関係者全員で合意し、開発の目的と範囲を明確にします。  
要件定義では、「何を作るか」を定義します。  
「どのように実現するか」は基本設計以降で検討します。  
  
# Operation  
要件定義書は、プロジェクト開始時に作成して終わる文書ではありません。  
要件変更や設計・開発の進行に合わせて継続的に見直し、常に最新の状態を維持します。  
要件を変更した場合は、内容を更新するとともに、第三者によるレビューを実施し、関係者間で内容を確認・合意した上で要件定義書へ反映します。  
要件定義書は、プロジェクト全体の共通認識を維持するための基準文書として管理します。  
  
# Content  
本書では、以下の項目について整理します。  

```
　凡例
　　🟥 必須：要件定義で必ず記載すべき項目  
　　🟨 推奨：プロジェクト規模に応じて記載  
　　🟩 必要に応じて：あるとなお良い
```
　🟥システム化の目的  
　　システムを導入する目的を記載します。  
　　箇条書きで良いので、明確に定義します。  
　🟥現状業務（As-Is）  
　　現在の業務フローを記載します。  
　　・業務内容を図形化して分かりやすく定義します。  
　　・使用システムも目的と実態を整理します。  
　　・手作業は内容とおおよその工数も記載します。    
　🟥業務課題  
　　現在の問題点を列挙します。  
　　問題点のみならず問題となっている理由も記載します。  
　🟥システム化後の業務（To-Be）  
　　理想とする状態を明確にします。  
　　実現可否は別途議論対象とします。  
　🟥対応範囲  
　　システムの適用範囲を明確に定義します。  
　　開発に従い、要件変更時に更新し、共有します。  
　🟥対応対象外  
　　明確に定義します。  
　　将来的に想定される機能も記載しておきます。  
　🟥利用者  
　　利用者によって、必要となる項目が変化するので、限定する必要があります。  
　　利用者のみならず、管理者、運用担当、保守担当にも着目して記載します。  
　🟥業務フロー  
　　旧 ⇒ 新で変化点を分かりやすく整理します。
```mermaid
flowchart LR;
    旧状態-->途中経過;
    途中経過-->新状態;
```
　🟥機能要件  
　　必要機能を列挙します。  
　🟥非機能要件  
　　下記項目を明確に定義します。  
　　　・性能  
　　　・可用性  
　　　・保守性  
　　　・セキュリティ  
　　　・バックアップ  
　　　・監視  
　　　・ログ  
　　　・レスポンス  
　　　・同時利用人数  
　　後々の問題とならない様、厳格に規定します。  
　🟥帳票一覧  
　　システムが出力する文書を定義します。  
　　　・見積書  
　　　・請求書  
　　　・納品書  
　　　・売上一覧  
　　　・CSV  
　　　・PDF  
　　　・Excel出力  
　　他、必要に応じて追加します。  
　🟥外部システム連携  
　　社内、社外、市販ツールを明記します。  
　　内容には連携方法、API、CSV、FTP、DB、メール等も含めます。  
　🟥制約事項  
　　適用範囲及び範囲外への対応を記載します。  
　　OS、ブラウザ、DB、クラウド、ライセンス、予算、納期等、分野別に整理します。  
　🟥前提条件  
　　運用条件、システム要件、運用規則を記載します。  
　🟥リスク  
　　予想できる障害及び障害時対応策(Contingency Plan)を記載します。  
　　影響度、発生確率、対策、担当者まで記載できると後々の指針となります。  
　🟥役割・責任  
　　各工程において、作成、レビュー、承認に対する権限を明確化します。  
　🟨マスタ一覧  
　　用途別に設定します。  
　　　・社員マスタ  
　　　・商品マスタ  
　　　・部署マスタ  
　　　・取引先マスタ  
　　他、必要に応じて追加します。  
　🟨優先順位  
　　優先順位を  
　　　Must > Should > Could > Won't  
　　に分けて記載すると分かりやすくなります。  
　🟨要件変更履歴   
　　要件ではありませんが、変更履歴は記すべきです。  
　　内容は、Version、更新日、更新者、変更内容等です。  
　🟩用語集  
　　意識統一のための用語を定義しておきます。  
　　受注、案件、案件番号、PJ、顧客、工番など自社固有の用語などもありますので、明記する事が重要です。
